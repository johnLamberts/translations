Background

The platform uses the Google Translate Script API for all language switching, supporting English (default), Korean, Japanese, and Chinese (Mandarin).

The client's new requirement: if a user types content in Chinese, that content should not be re-translated back into Chinese when another Chinese-speaking viewer reads it. The same logic applies to Korean, Japanese, and English — translation should only fire when the content language differs from the viewer's selected language.

⚙️

The Decision: Continue with Google Translate

Rather than switching engines, this path makes four targeted adjustments to how and when Google Translate fires — satisfying the new requirement without a system overhaul.

Required Changes — 4 Modifications
1

Lock the App to 4 Languages Only

Restrict changeLanguage() to only accept EN, KO, JA, ZH. Any other language code is rejected or ignored. Update the Language type and localStorage validation to match.

Why this matters Currently the system validates against 7 languages including vi, th, and es — none of which the client uses. Tightening this scope prevents accidental language drift.
2

Add a Same-Language Skip Rule

Before Google Translate processes user-typed content, the system will detect what language that content is written in. If it already matches the viewer's language — skip translation and show the content as-is.

Why this matters This is the core fix. A Korean user's input won't be re-translated when a Korean viewer reads it. Same for Chinese, Japanese, and English.
3

Protect All Input Fields Consistently

User-typed fields — names, job descriptions, free-text — must never be auto-translated by Google. Currently this is handled by 2,782 manually-placed notranslate tags across 305 files, with no consistent pattern or safety net.

Why this matters The current approach has inconsistencies and misplacements already in the codebase. This pass corrects them all — though without automation, it requires careful manual review.
4

Stabilise Language-Switching Behaviour

Switching directly between two non-English languages (e.g. JA → KO) causes Google Translate to sometimes get stuck. The existing bounce-through-English technique already handles this in changeLanguage() and must be preserved.

Why this matters Without the bounce technique the UI can appear frozen mid-switch — a visible UX failure during language selection.

⚠️ What This Path Does Not Solve

  • Google's DOM injection remains. Google Translate injects an iframe into the page. Scroll jumps, hydration warnings, and React re-render glitches are not fixed by this path.
  • Translation quality is Google's. Job titles, industry terms, and HR-specific language depend entirely on Google's engine. Mistranslations have no override mechanism.
  • No safety net without ESLint. Any new component added after this work can silently re-introduce a raw notranslate error or miss it entirely. The 2,782-element problem will grow back over time.
  • These are mitigations, not a full fix. The changes satisfy the client's current requirement, but the structural limitations of the Google Translate script integration remain.
Engineering Note

This path works — with a clear boundary.

All four changes can be implemented and will satisfy the client's current requirement. The language-aware skip logic is achievable within the existing Google Translate setup.

This is a targeted fix, not a foundation. If requirements grow — more languages, per-field behaviour, or cost concerns — the team recommends moving to a self-hosted translation engine at that point. For now, this approach is practical and ships in a defined timeframe.

Deep Dive — The notranslate Problem

The largest practical challenge in this path is the notranslate class currently spread manually across 2,782 elements in 305 files. This section explains what it does, why it got out of hand, and exactly what needs to change — with code examples.

What Is notranslate and Why Does It Exist?

When Google Translate is active, it attempts to translate everything visible in the DOM — including user-typed input values, form fields, and profile data. The official Google workaround is class="notranslate" on any element you want skipped. This was applied manually, one file at a time, across the entire codebase — with no central control, no pattern, and no way to enforce it on new components.

What the Current Code Looks Like

notranslate is scattered inconsistently — on inputs, wrappers, display fields — with no shared rule about which element it belongs on.

The problem isn't just the volume — it's that there's no shared rule. Some files protect the input, some protect the wrapper div, some do both, and some new components miss it entirely. One existing bug in JobDetail.tsx: notranslate was placed on the SunDim icon instead of the text next to it — the wrong element entirely.

What Needs to Change — Step by Step
Step 1

Create a shared <NoTranslate> wrapper component

+

Instead of 305 files each manually writing className="notranslate", create one shared component that owns the logic. All 305 files then import and use it — so any future change to the approach only requires editing one file.

components/NoTranslate.tsx — create once ✅ New 
// Renders as any HTML tag, always applies notranslate
export function NoTranslate({ children, as: Tag = "span", className, ...props }) {
  return (
    <Tag
      className={["notranslate", className].filter(Boolean).join(" ")}
      {...props}
    >
      {children}
    </Tag>
  );
}

// Usage — replaces every manual className="notranslate":
// <NoTranslate>{userInput}</NoTranslate>
// <NoTranslate as="div"><input value={...} /></NoTranslate>
Step 2

Manually update 2,782 elements across 305 files

+

With no codemod, this is done by two developers working through the files in parallel. Each element needs to be reviewed in context — not just find-and-replaced — because some instances are misplaced (like the SunDim bug) and need correcting, not just renaming.

The work involves three types of changes across the 305 files:

Each file also needs the import added at the top:
import { NoTranslate } from "@/components/NoTranslate";

Step 3

Add smart language detection inside <NoTranslate>

+

Once all 305 files use <NoTranslate>, the client's same-language skip rule can be added to that one component — and it propagates everywhere automatically. The component reads the current UI language, detects the content's language, and only applies notranslate when they match.

components/NoTranslate.tsx — upgraded with detection ✅ Updated 
export function NoTranslate({ children, as: Tag = "span", className, ...props }) {
  const { language } = useTranslation();
  const text = typeof children === "string" ? children : "";
  const detectedLang = detectLanguage(text);

  // Only protect when content language matches the UI language
  const protect = !text || detectedLang === language;

  return (
    <Tag
      className={[protect ? "notranslate" : "", className].filter(Boolean).join(" ")}
      {...props}
    >{children}</Tag>
  );
}

Pros & Cons — Staying on Google API

A clear-eyed assessment based on the actual codebase, not just theory.

✅ Pros
✓
No engine switch — predictable scope
No new infrastructure to deploy. The work is well-understood: find the elements, update them, verify. Effort is large but the type of work is familiar for the dev team.
✓
Client requirement is fully satisfied
The same-language skip logic directly addresses the ask — ZH input won't be re-translated for ZH viewers. Same for KO and JA.
✓
Existing local translation maps are untouched
The tIndustry, tSkill, tJob, tLanguage and all other local lookup functions continue working as-is — zero regression risk on dropdown translations.
✓
Codebase is audited and understood after this work
Going through 305 files manually means the team will understand exactly what is and isn't protected. This audit knowledge has value — even if it's painful to acquire.
✓
Sets up a clean foundation for future migration
Once <NoTranslate> is in place, switching to LibreTranslate later only requires changing that one component. The 305 files won't need touching again.
❌ Cons
✗
High manual effort — 29–37 man-days
There is no automation. 2,782 elements across 305 files touched by hand. Each one needs context, judgment, and verification — not a simple find-and-replace.
✗
No safety net — errors will recur without ESLint
Any new component added after this work can silently re-introduce a raw notranslate mistake. The 2,782-element problem will grow back over time with no guard in place.
✗
DOM injection issues remain
Google Translate still injects an iframe into the page. Scroll jumps, hydration warnings, and React re-render conflicts are not fixed by this path.
✗
Language detection adds a new dependency
Detecting content language requires a lightweight library (e.g. tinyld). Short strings — 1 or 2 words — are notoriously unreliable to detect correctly.
✗
No control over translation quality
Job titles, industry terms, and HR-specific language depend entirely on Google's engine. Mistranslations in professional contexts have no override mechanism.
