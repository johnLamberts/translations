Background

The platform currently uses the Google Translate Script API to handle all language switching across the application. The system supports four languages: English (default), Korean, Japanese, and Chinese (Mandarin).

The client's new requirement is straightforward but important: if a user types content in Chinese, that content should not be re-translated back into Chinese when another Chinese-speaking user views it. The same applies to Korean, Japanese, and English — the app should be smart enough to skip translation when the content language already matches what the viewer is reading in.

⚙️

The Decision: Continue with Google Translate

Rather than switching to a different translation engine, this path keeps Google Translate as-is and makes targeted adjustments to how and when it fires — to satisfy the new language-aware requirement. These are four specific changes needed in the system.

Required Changes — 4 Modifications
1

Lock the App to 4 Languages Only

Right now, the system technically accepts any language code. We need to explicitly restrict it to only English, Korean, Japanese, and Chinese. Anything outside those four should be ignored.

Why this matters Google's widget, left unconfigured, can show a dropdown with 100+ languages. This scopes it to exactly what the client's users need.
2

Add a "Same-Language Skip" Rule

Before Google Translate processes any user-typed content (like a name, bio, or job description), the system will first check what language that content is written in. If it's already in the same language the viewer is using — skip translation entirely and show it as-is.

Why this matters This is the core fix for the client's requirement. A Korean user's input won't get re-translated when a Korean viewer reads it.
3

Protect All Input Fields from Auto-Translation

User-typed fields (resume text, names, addresses, free-text answers) should never be auto-translated by Google unless explicitly needed. We will apply a consistent, system-wide protection to all input areas — replacing the current fragmented approach across 305 files.

Why this matters The current setup has inconsistent protection. This unifies it so no new input field ever accidentally gets mis-translated in the future.
4

Control How Language Switching Behaves

Switching from one foreign language directly to another (e.g. Japanese → Korean) currently causes Google to sometimes get stuck or behave unexpectedly. A routing adjustment will be added: always pass through English as an intermediate step when switching between two non-English languages.

Why this matters Prevents the UI from appearing broken or frozen mid-switch — important for user experience.

⚠️ What This Path Does Not Solve

  • Google's DOM injection remains. The way Google Translate works — by injecting code directly into the page — can still occasionally cause visual glitches or unexpected behavior in certain parts of the app. These are not new issues, but they are not fixed by this path.
  • Translation quality is Google's. We have no control over how accurately Google translates industry-specific terms, job titles, or nuanced phrases. Quality is fully dependent on Google's engine.
  • Cost and usage limits. Google Translate's free tier has usage caps. As the platform scales, charges may apply. There is currently no usage monitoring in place.
  • These are mitigations, not a full fix. The changes above make the system behave correctly for the client's new requirement, but they don't address the underlying structural limitations of the Google Translate script integration.
Engineering Note

This path works — with a clear boundary.

All four changes above can be implemented and will satisfy the client's current requirement. The language-aware skip logic is achievable within the existing Google Translate setup.

However, it is worth understanding that this is a targeted fix, not a foundation. If requirements grow — more languages, more complex per-field behavior, or cost concerns — the team recommends revisiting a move to a self-hosted translation engine at that point. For now, this approach is practical and can ship quickly.

Deep Dive — The notranslate Problem

One of the largest practical challenges in this migration is the notranslate class currently spread across 305 files. This section explains what it does, why it got out of hand, and exactly what needs to change — with lightweight code examples for the engineering team.

What is notranslate and Why Does It Exist?

When Google Translate is active on a page, it attempts to translate everything it can see in the DOM — including content that should never be touched, like user-typed input values, form fields, dropdowns, and profile data.

To stop Google from touching specific elements, the official Google workaround is to add class="notranslate" to any HTML element you want protected. Google Translate reads this class and skips that element entirely.

The problem: this was applied manually, one file at a time, across the entire codebase. Over time it grew to 305 files with no central control — meaning every new component risks missing it, and removing or changing the approach requires touching all 305 files again.

What Needs to Change — 4 Steps

Click each step to expand the explanation and see the code pattern involved.

Step 1

Create a shared <NoTranslate> wrapper component

+

Instead of each of the 305 files manually adding className="notranslate", we create a single shared component that does it in one place. Every file then uses this component instead — so if we ever need to change the approach (e.g. switch away from Google), we only change one file, not 305.

components/NoTranslate.tsx — new shared component ✅ New (create once) 
// Create this file ONCE. Every other component imports and uses it.
export function NoTranslate({
  children,
  as: Tag = "span",
  className,
  ...props
}: {
  children: React.ReactNode;
  as?: keyof JSX.IntrinsicElements;
  className?: string;
}) {
  return (
    <Tag
      className={["notranslate", className].filter(Boolean).join(" ")}
      {...props}
    >
      {children}
    </Tag>
  );
}

// Usage in any component:
// <NoTranslate>{userInput}</NoTranslate>
// <NoTranslate as="div"><input value={...} /></NoTranslate>

The as prop lets it render as any HTML tag — a span for inline text, a div for input wrappers — so it handles all the different use cases currently scattered across the 305 files.

Step 2

Run an automated codemod to replace all 305 manual usages

+

With the <NoTranslate> component in place, we run a codemod script — an automated code-transformation tool — that finds every instance of notranslate across the codebase and rewrites it to use the new component. This avoids manual changes to 305 files one by one.

What is a codemod? Think of it like a smart find-and-replace that understands code structure, not just text. It can tell the difference between a class on a div vs. a class on an input and apply the right transformation to each case.
scripts/codemod-notranslate.ts — automated replacement Run once 
// What the codemod does — conceptually:

// BEFORE (found in 305 files, in various forms):
<span className="notranslate text-sm">{value}</span>
<div  className="notranslate"><input .../></div>
<input className="notranslate border rounded" />

// AFTER (uniform, centrally controlled):
<NoTranslate className="text-sm">{value}</NoTranslate>
<NoTranslate as="div"><input .../></NoTranslate>
<NoTranslate as="input" className="border rounded" />

// The codemod also adds the import at the top of every affected file:
import { NoTranslate } from "@/components/NoTranslate";

After the codemod runs, the word notranslate should appear in only one place in the entire codebase: inside the NoTranslate.tsx component itself.

Step 3

Add smart language-detection inside NoTranslate for the new requirement

+

The new client requirement — "don't translate content that's already in the viewer's language" — can now be handled in the single NoTranslate component rather than in each of the 305 files. The component reads the current UI language and the detected language of the content, then decides whether to apply notranslate dynamically.

components/NoTranslate.tsx — upgraded with smart detection ✅ Updated 
import { useTranslation } from "@/context/translation-provider";
import { detectLanguage } from "@/utils/detectLanguage";

export function NoTranslate({ children, as: Tag = "span", className, ...props }) {
  const { language } = useTranslation();

  // Detect what language the content itself is written in
  const contentText = typeof children === "string" ? children : "";
  const contentLang = detectLanguage(contentText);

  // If content language matches UI language → protect it (notranslate)
  // If content language differs            → let Google translate it
  const shouldProtect = contentLang === language || contentText === "";

  return (
    <Tag
      className={[
        shouldProtect ? "notranslate" : "",
        className
      ].filter(Boolean).join(" ")}
      {...props}
    >
      {children}
    </Tag>
  );
}

Because all 305 files now use <NoTranslate>, this one change to the component propagates the new behavior everywhere automatically — no need to touch each file again.

Step 4

Prevent future leaks — make notranslate a lint error on raw elements

+

To prevent any future developer from accidentally writing className="notranslate" directly on an element again, we add a linting rule that flags it as an error during development. This enforces that all protection must go through the <NoTranslate> component.

.eslintrc — custom rule to prevent raw notranslate usage Add to CI/CD 
// In your ESLint config — flags any raw "notranslate" className
// outside of the NoTranslate component itself
{
  "no-restricted-syntax": [
    "error",
    {
      "selector": "JSXAttribute[name.name='className'] > Literal[value=/notranslate/]",
      "message": "Use <NoTranslate> instead of className='notranslate' directly."
    }
  ]
}

// What the developer sees if they add it manually:
// ❌  ESLint: Use <NoTranslate> instead of className='notranslate' directly.
//     at ProfileCard.tsx:42

This makes the fix self-reinforcing — the codebase can never silently drift back to the old pattern. Any new component that needs protection is forced to use the right approach.

Pros & Cons — Staying on Google API

A clear-eyed assessment based on the actual code — not just theory.

✅ Pros
✓
No engine switch — zero migration risk
All existing Google infrastructure, cookies, and the translation-provider.tsx stays mostly intact. No unfamiliar system to learn or deploy.
✓
Faster to ship — 2–3 days vs. weeks
The codemod handles the 305-file problem automatically. The <NoTranslate> wrapper is a small addition, not a system overhaul.
✓
Client requirement is fully satisfied
The same-language skip logic directly addresses what the client asked for — ZH input won't be re-translated for ZH viewers, same for KO and JA.
✓
Existing local translation maps are untouched
The tIndustry, tSkill, tJob, tLanguage and all other local lookup functions continue working as-is — no regression risk on dropdown translations.
✓
Sets up a clean foundation for future migration
Once <NoTranslate> is in place and the codebase is centralized, switching to LibreTranslate later only requires changing that one component — the 305 files don't need touching again.
❌ Cons
✗
DOM injection issues remain
Google Translate still injects an <iframe> into the page. Existing glitches — scroll jumps, hydration warnings, React re-render conflicts — are not fixed by this path.
✗
Language detection adds a new JS dependency
To detect "is this text Chinese or Korean?" we need a lightweight library (e.g. tinyld or franc). It adds ~10–20kb and introduces a dependency to maintain. Short strings (1–2 words) are also notoriously hard to detect reliably.
✗
Bounce technique is still required
Switching JA → KO still needs the EN-bounce workaround already in changeLanguage(). This is a Google limitation — not fixable without changing engines.
✗
Codemod output requires manual verification
Automated transforms across 305 files will have edge cases — e.g. JobDetail.tsx has 15+ separate notranslate usages across mixed element types. Manual review is still needed after the codemod.
✗
No control over translation quality
Job titles, industry terms, and HR-specific language depend entirely on Google's engine. Mistranslations in professional contexts have no override mechanism.
Real Code Analysis — JobDetail.tsx

JobDetail.tsx is a good representative of the problem at scale. It mixes locally-translated values (via tIndustry, tSkill, etc.) with raw user-input data (job title, company name, location) — and both types have notranslate applied, but inconsistently.

🔍

15+ separate notranslate usages were found in JobDetail.tsx alone. They are applied directly on h2, span, p, and Badge elements — with no consistent pattern. One element even has it on the icon wrapper (SunDim) instead of the text next to it.

Issue A

User data fields — job title, company name, location

+

These are raw server values — they were typed by an employer and should never be auto-translated. Currently protected by notranslate directly on the element. Under the new requirement, these should always be protected regardless of language — a job title is a proper noun and must never be changed.

JobDetail.tsx — lines 113–119 (current) ❌ Raw notranslate 
// Job title — always a proper noun, never translate
<h2 className="text-2xl font-extrabold ... notranslate">
  {jobDetail.title}
</h2>

// Company name
<span className="notranslate">
  {jobDetail.company?.name}
</span>

// After: use NoTranslate — always protected, no language detection needed
<NoTranslate as="h2" className="text-2xl font-extrabold ..." alwaysProtect>
  {jobDetail.title}
</NoTranslate>

The alwaysProtect prop tells <NoTranslate> to skip language detection entirely and always apply notranslate — appropriate for proper nouns and employer-entered data.

Issue B

Locally-translated values — tIndustry, tSkill, tLanguage, tEmpType

+

These values come from the local translation maps already in translation-provider.tsx — they are already correctly translated into the UI language before they reach the DOM. Google Translate must be blocked from touching them, or it will double-translate.

JobDetail.tsx — industry / skill / empType badges (current) ⚠️ Inconsistent 
// tIndustry() already returns the correct language — Google must not touch this
<p className="font-semibold text-sm notranslate">
  {tIndustry(jobDetail.company?.industry)}
</p>

// tSkill() — some badges have notranslate, some don't ❌
<Badge variant="outline" className="bg-muted/50">  // ← missing notranslate!
  {skill.skillName && tSkill(skill.skillName)}
</Badge>

// After: NoTranslate with alwaysProtect — pre-translated values never need Google
<NoTranslate as="p" className="font-semibold text-sm" alwaysProtect>
  {tIndustry(jobDetail.company?.industry)}
</NoTranslate>

<NoTranslate alwaysProtect>{skill.skillName && tSkill(skill.skillName)}</NoTranslate>

Notice the skill badge in the current code is missing notranslate entirely — this is exactly the kind of leak that happens with the manual approach, and why the centralized <NoTranslate> + lint rule matters.

Issue C

The misplaced notranslate on the SunDim icon

+

One specific instance in JobDetail.tsx shows the risk of manual placement: notranslate was accidentally put on the icon (SunDim) instead of the adjacent text (jobDetail.daysOff). The icon doesn't need protection — the text does.

JobDetail.tsx — days off row (current) 🐛 Misplaced 
// notranslate is on the ICON — wrong element! The text has no protection.
<div className="flex items-center">
  <SunDim className="mr-2 h-10 notranslate" />  // ← icon doesn't need this
  <p className="font-semibold text-sm">       // ← text is unprotected ❌
    {jobDetail.daysOff}
  </p>
</div>

// After: protection is on the right element
<div className="flex items-center">
  <SunDim className="mr-2 h-10" />
  <NoTranslate as="p" className="font-semibold text-sm" alwaysProtect>
    {jobDetail.daysOff}
  </NoTranslate>
</div>

This kind of bug is invisible at first glance and would only surface at runtime when Google Translate is active — exactly the environment where it's hardest to catch. The codemod won't catch this automatically either; it requires the manual review pass.
